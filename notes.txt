Significance rules and calculations.
Kalvin employs an idea of significance to choose between two klines with one being more or less significant that another with respect to the current state of the model. Significance is a calculated value ranging from S1 (fully significant) through to S4 (fully insignificant). 

The calculation is arrived as a metric distance between the nodes of two klines. Given a query that selects two candidate answers, the answer closer to the query is the more significant.

The same calculated value is also a measure of a kline's structural entropy. More work is required to move a kline from a low significance to a high one. 




It is calculated as the measure of a kline's structural entropy. A highly significant kline is one where the entropy level is very low, and conversely a highly insignificant kline is one where the entropy level is very high.

A signature that is closer to its nodes has lower entropy than one that is further away. Significance is expressed as the inverse of the entropy calculation so that for any two klines, it is possible to state which is the more significant through simple arithmetic.

The following tables demonstrate how significance is the inverse of the metric distance between significance and a set of nodes. Thus Significance is higher when signature and nodes are closer together.

Variables used:
S1..S4 ~> The significance of a KLine expressed as an bit pattern where S1 tends to all bits set and S4 tends to all bits clear. Note that this final value is the inverse of the calculated distance between a signature and its nodes.

n, m, F, 0 ~> bit patterns that overlay the calculated significance:
  n can be any value between 1 and 0xF
  m is the inverse of n after significance has been fully inverted

  For example if the calculated distance between a sig and its nodes is 0xABC, its significance will be 0x543

### The Grounding Principle
A KLine is grounded (S1) if all of its nodes are grounded. Literal nodes are implicitly grounded, but non literal nodes can only be grounded through reciprocal kline reference (countersigning): KLines A and B reference each other through their nodes.

The relationship between a Signature and its Nodes can be viewed as a measure of its structural entropy. A signature that is closer to its nodes has lower entropy than one that is further away. A signature's closeness to its nodes is a measure of entropy. A signature that has more information than its nodes suggest is over-fitting, and the system will try to balance this by substituting klines with higher entropy with klines of lower entropy.

 Proposed s3_distance Semantics

 Context: S3 fires when zero query nodes appear in the candidate's node sequence — no direct membership match at all.

 ### Starting Distance

 D_BOUNDARY — all S3 bits clear. Distance grows upward toward D_MAX with mismatch and discovery (same philosophy as S2 starts at 0).

 ### Algorithm — Edge Hops for Both Sides

 For each node in query and candidate, compute edge_hops(node):
 - hops = 0: node is unresolvable (dead end) — maximum S3 penalty
 - hops = MAX_HOP: node hit the chain limit — also heavy penalty
 - 0 < hops < MAX_HOP: node IS reachable via the graph — this is key

 ### Promotion Rule (S3 → S2)

 If any node (query or candidate) has 0 < hops < MAX_HOP, the pair has graphical connectivity — there IS a path from the node to grounded knowledge. This promotes the hop distance into S2 range rather than S3. The reasoning: if edges
 resolve, the relationship isn't purely connotational any more.

 ### Match Rule (Stays S3 with computed distance)

 If a query node's hop chain converges with a candidate node's hop chain — i.e., both resolve to the same canonical kline — the distance stays in S3 range but is computed as:

 ```
   distance = q_hops + c_hops
 ```

 This captures "they're related, but indirectly — through the graph rather than by direct membership."

 ### No Connectivity (Pure S3)

 If ALL nodes have hops = 0 or MAX_HOP, and no convergence targets match → D_MAX - 1 (worst S3 distance). Purely connotational.

Notes:
The current system can choose between two (or more) tokenizers, each of which has its own internal strategy for token encoding|decoding. However the chosen strategy is not totally opaque to Kalvin because of the way that significance is generated between a query and a set of candidates. In particular Kalvin must operate at the bit level to generate fine grained candidate selection that directly informs significance routing. This means that all tokenization schemes must fit the same structural constraints, and through this, must encode the same kind of bit level information even when the vocabularies from which the tokens are generated ace completely incompatible.

The Mod(N)Tokenizer scheme aims to encode single ASCII characters as bit positions. The implication of this is that the vocabulary size is obviously limited to the bit range supported by the token (32, 64 or 128). With such a small vocabulary it is not a tokenizer responsibility to encode or compress external information. In fact this tokenizer class reserves a single bit as an escape mechanism for encoding literal string values directly into tokens. Kalvin understand this convention well enough to isolate these token values when dealing with significance. 

The Mod(N)Tokenizer class is designed to work directly with KScript. Its purpose is research. Its limited vocabulary leads to compact, observable and accessible knowledge graphs that can be reasoned with and even hand crafted in the pursuit of Kalvin development. MHALL is a common entry point for rationalisation, and it consists of just 4 (not 5) tokens that fit nicely into a single signature. From this accessible starting point we can explore how to build Kalvin's agency to the point that we know that it knows who Mary is, and what it means to have had a little lamb.

The BPE Tokenizer (not yet completed) is designed for more robust and granular rationalisation of much larger knowledge graphs. And yet it must fit the same token constraints that allow Kalvin to work with signatures in the first place. Therefore it is not technically a BPE tokenizer. Its aim is to encode token bit structures that operate exactly the same way as Mod tokens do. It has the freedom to encode these structures from trained BPE datasets so that the underlying encode|decode strategy operates on BPE sequences rather than single ASCII characters, but once encoded the only difference is one of granularity.

In the Kalvin system, the granularity of a token is a measure of dimensionality. In the ModNTokenizer, this dimensionality is very course and quite wasteful amounting to a single dimension reserved for each encoded ASCII character (also allowing for character collisions). In the BPS Tokenizer, the same dimensionality is populated by a trained nlp part-of-speech encoding. In both schemes the purpose is not to encode knowledge directly but to encode granular dimensionality that kalvin can use to construct knowledge as a graph.

The training loop concerns a teacher (human or automated) and a kalvin instance. The teacher creates training scripts (using KScript) that compile down to klines: a query and an expectation with optional intermediate nested query/expectations as required. The teacher delivers these to kalvin through the rationalisation API, and subscribes to kalvin's event bus to receive submissions.

A teacher 
  ~> controls learning rate by raising or lowering temperature
  ~> controls learning effort through adjustments to top_p and top_k
  ~> constructs kline queries and expectations
  ~> encodes and delivers kline queries to kalvin
  ~> receives submitted kline events from kalvin
  ~> compares and contrasts event klines against expectations
  ~> countersigns matching klines

Kalvin 
  ~> accepts and rationalises kline queries
  ~> cogitates where understanding is lower than temperature dictates
  ~> emits significant events guided by temperature
  ~> promotes countersigned queries

KScript can be used to construct suitable training scripts that encode queries, intermediate steps and expectations. 

Each submitted training script initiates a new frame. The teacher uses this information to correlate and coordinate immediate and delayed events.

The teacher feedback loop is synchronised by rationalisation response: True=immediate, False=cogitating. The teacher uses this response to coordinate event frames.

A training schedule.
A training loop pipeline can be established fairly easily and would support the growth of a kalvin knowledge graph through reentrant rationalisation. The @docs/training-loop.md document describes how this might be achieved and sets the stage for implementation (via appropriate specs and planning).

However, a more detailed exploration of the training schedule will be useful for developing a consistent set of training data. The first kalvin knowledge base will be empty (a clean slate), and so it will be important to script and deploy various countersigned klines appropriate to the bootstrapping of Kalvin towards a specific knowledge domain. 

For example, in the MHALL scenario, it might be appropriate to deploy various countersigned grammar related klines so that Kalvin is even able to entertain rationalising that famous nursery rhyme. Or it might be sufficient for a fully interactive human-kalvin training session in which the human endorses Kalvin's own efforts for it to learn to understand syntax. The interesting point here is that these two approaches are not mutually exclusive, and the trainer is responsible for the overall management of training data - what to create, when to deploy it, and so on. This is the basic idea of a training schedule (perhaps training curriculum is a more appropriate term).


When does a training schedule complete?
The simple answer is that it completes when Kalvin has grounded enough of the information to satisfy the trainer. But Kalvin is not designed to operate in a training mode vs inference mode kind of way. For Kalvin, there is no training, but there is always learning. Kalvin recognises what it doesn't know when it reacts to novel information. So in an important sense from the management of a Kalvin instance perspective, training never really stops.

Here, the idea of a training loop can now be seen as (or replaced by) an operational harness. Under the supervision of a capable harness, Kalvin might be encouraged to emit S2 for partial understanding. The harness is now acting as a mentor. It is programmed to handle the possibility of Kalvin baulking at novel information, so it is capable of notifying the organisation of the knowledge gap. It might be prudent to update Kalvin to close the gap: effectively training Kalvin on the job.

terminology recap:
- proposal: the kline selected through cogitation emitted to the subscribed teacher
- overfitting: A kline overfits when its signature provides less information than its nodes (additional noisy nodes)
- underfitting: A kine underfits when its signature provides more information than its nodes ((missing signal nodes)

When the signature of a kline fully describes its nodes, it is called an S1 canon - the signature leads to the fully (and correctly) grounded set of nodes that encode some concept. Any mismatch between signature and nodes results in a lower S2 significance, either because of over-fitting (too many nodes) or under-fitting (too few nodes)

Conceptually, S2 represents a graded idea of understanding. As a system learns to ground more and more of a kline's nodes, so its significance rises towards S1. Therefore it is useful to be able to say "I understand some or even most of this kline" as impetus for learning more, or for proposing alternative or partial ideas where temperature encourages it. In short, S2 leads to agency, the ability and sovereignty to make choices that underpins the purpose of a kalvin system.

The current system recognises miss-fitting but does not act on it. When an S2 kline fails the countersignature test, kalvin simply moves on to the next kline candidate in the list. This suggests that S2 klines are somehow incomplete and will only become useful when fully canonised.

Before kalvin can be fully operational it needs to expand its cogitation abilities by expanding or contracting S2 klines to match their signatures. These klines belong to the set of candidate klines passed thought to the cogitator pipeline. After failing to establish countersignature, the cogitator will try to establish if underfit or overfit cogitation applies. 

This additional S2 processing captures the essence of self study. This is where kalvin might supply missing nodes, or remove redundant ones to better fit a signature. Under cogitation, it is the signature that is invariant and it is the nodes in this temporary working state that must change to match it. 

The resulting kline is just a proposal, but by invoking the outside opinion of a mentor, it might generate alternative paths to S1 ratification. This is where the true power of cogitation takes hold. But there are constraints:
- Kalvin is not allowed to throw away data. If it removes nodes then it must find a way to use them somewhere else. The redundant nodes must form a new kline or must complete some other underfitting kline.
- Kalvin is not allowed to invent new data. If it adds nodes then they must already exist as klines in the model. 
- Kalvin is allowed to replace whole subsets of nodes with different sized subsets provided it does not break either of the above rules.

These cogitate operations allow kalvin to reshape a kline to better fit a signature and through this the ability to reason far more deeply than a simple pattern matching system can achieve.

Important spec change for countersigned definition and code changes for model.is_countersigned and model._is_countersigned_in_model:
A kline is countersigned if its nodes_signature exists as a countersigning kline with one node - the countersigned kline's signature.

    Query = {Q: [A, B]}
    Countersigner = {AB: [Q]}

Note that both query and countersigning signatures can represent multiple klines. Countersigning klines are those that have one node that references the countersigned kline.

This has important consequences for ratification. It means that ratification can be tested in the fast lane in rationalisation before any candidates are established.

It also means that is_countersigned function is no longer required.

```new is_countersigned
def is_countersigned(self, kline: KLine) -> bool:
  nodes_signature = make_signature(kline.nodes)
  for countersigned in self.find_all(nodes_signature):
    if len(countersigned.nodes == 1) and countersigned.nodes[0] == kline.signature:
      return True
  return False
```
