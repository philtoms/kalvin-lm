Significance rules and calculations.
Kalvin employs an idea of significance to choose between two klines with one being more or less significant that another with respect to the current state of the model. Significance is a calculated value ranging from S1 (fully significant) through to S4 (fully insignificant). 

The calculation is arrived as a metric distance between the nodes of two klines. Given a query that selectes two candidate answers, the answer closer to the query is the more significant.

The same calculated value is also a measure of a kline's structural entropy. More work is required to move a kline from a low significance to a high one. 




It is calculated as the measure of a kline's structural entropy. A highly significant kline is one where the entropy level is very low, and conversely a highly insignificant kline is one where the entroy level is very high.

A signature that is closer to its nodes has lower entropy than one that is further away. Significance is expressed as the inverse of the entropy calculation so that for any two klines, it is possible to state which is the more significant through simple arithmetic.

The following tables demonstrate how significance is the inverse of the metric distance between significance and a set of nodes. Thus Significance is higher when signature and nodes are closer together.

  n can be any value between 1 and 0xF

  For example if the calculated distance between a sig and its nodes is 0xABC, its significance will be 0x543

 Proposed s3_distance Semantics

 Context: S3 fires when zero query nodes appear in the candidate's node sequence — no direct membership match at all.

 ### Starting Distance

 D_BOUNDARY — all S3 bits clear. Distance grows upward toward D_MAX with mismatch and discovery (same philosophy as S2 starts at 0).

 ### Algorithm — Edge Hops for Both Sides

 For each node in query and candidate, compute edge_hops(node):
 - hops = 0: node is unresolvable (dead end) — maximum S3 penalty
 - hops = MAX_HOP: node hit the chain limit — also heavy penalty
 - 0 < hops < MAX_HOP: node IS reachable via the graph — this is key

 ### Promotion Rule (S3 → S2)

 If any node (query or candidate) has 0 < hops < MAX_HOP, the pair has graphical connectivity — there IS a path from the node to grounded knowledge. This promotes the hop distance into S2 range rather than S3. The reasoning: if edges
 resolve, the relationship isn't purely connotational anymore.

 ### Match Rule (Stays S3 with computed distance)

 If a query node's hop chain converges with a candidate node's hop chain — i.e., both resolve to the same canonical kline — the distance stays in S3 range but is computed as:

 ```
   distance = q_hops + c_hops
 ```

 This captures "they're related, but indirectly — through the graph rather than by direct membership."

 ### No Connectivity (Pure S3)

 If ALL nodes have hops = 0 or MAX_HOP, and no convergence targets match → D_MAX - 1 (worst S3 distance). Purely connotational.

Notes:
The current system can choose between two (or more) tokenizers, each of which has its own internal strategy for token encoding|decoding. However the chosen strategy is not totally opaque to Kalvin because of the way that significance is generated between a query and a set of candidates. In particular Kalvin must operate at the bit level to generate fine grained candidate selection that directly informs significance routing. This means that all tokenization schemes must fit the same structural constraints, and through this, must encode the same kind of bit level information even when the vocabularies from which the tokens are generated ace completely incompatible.

The Mod(N)Tokenizer scheme aims to encode single ASCII characters as bit positions. The implication of this is that the vocabulary size is obviously limited to the bit range supported by the token (32, 64 or 128). With such a small vocabulary it is not a tokenizer responsibility to encode or compress external information. In fact this tokenizer class reserves a single bit as an escape mechanism for encoding literal string values directly into tokens. Kalvin understand this convention well enough to isolate these token values when dealing with significance. 

The Mod(N)Tokenizer class is designed to work directly with KScript. Its purpose is research. Its limited vocabulary leads to compact, observable and accessible knowledge graphs that can be reasoned with and even hand crafted in the pursuit of Kalvin development. MHALL is a common entry point for rationalisation, and it consists of just 4 (not 5) tokens that fit nicely into a single signature. From this accessible starting point we can explore how to build Kalvin's agency to the point that we know that it knows who Mary is, and what it means to have had a little lamb.

The BPE Tokenizer (not yet completed) is designed for more robust and granular rationalisation of much larger knowledge graphs. And yet it must fit the same token constraints that allow Kalvin to work with signatures in the first place. Therefore it is not technically a BPE tokenizer. Its aim is to encode token bit structures that operate exactly the same way as Mod tokens do. It has the freedom to encode these structures from trained BPE datasets so that the underlying encode|decode strategy operates on BPE sequences rather than single ASCII characters, but once encoded the only difference is one of granularity.

In the Kalvin system, the granularity of a token is a measure of dimensionality. In the ModNTokenizer, this dimensionality is very course and quite wasteful amounting to a single dimension reserved for each encoded ASCII character (also allowing for character collisions). In the BPS Tokenizer, the same dimensionality is populated by a trained nlp part-of-speech encoding. In both schemes the purpose is not to encode knowledge directly but to encode granular dimensionality that kalvin can use to construct knowledge as a graph.
